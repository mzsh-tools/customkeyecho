# 输入驱动

## 需求

- 模拟键盘按键（按下 + 释放）
- 模拟鼠标按键（按下 + 释放）
- 尽可能被游戏识别为真实输入

## 设计方案

### InputDriver.py

通过 ctypes 调用 Win32 `SendInput` API，使用硬件扫描码模拟输入。

### 键盘输入

- VK 码通过 `MapVirtualKeyW` 转换为扫描码
- 使用 `KEYEVENTF_SCANCODE` 标志发送扫描码而非虚拟键码
- 扩展键（方向键、Insert/Delete、Win 键等）附加 `KEYEVENTF_EXTENDEDKEY` 标志
- 按下与释放之间 10ms 延迟模拟真实按键时长

### 鼠标输入

按键编号映射：

| Code | 按键 | Down 标志 | Up 标志 | MouseData |
|------|------|-----------|---------|-----------|
| 1 | 左键 | LEFTDOWN | LEFTUP | 0 |
| 2 | 右键 | RIGHTDOWN | RIGHTUP | 0 |
| 3 | 中键 | MIDDLEDOWN | MIDDLEUP | 0 |
| 4 | 侧键1 | XDOWN | XUP | XBUTTON1 |
| 5 | 侧键2 | XDOWN | XUP | XBUTTON2 |

### ctypes 结构体

`_INPUT` 结构体包含 `_INPUTUNION`（Union），内含 `_KEYBDINPUT` 和 `_MOUSEINPUT`。结构体大小 40 字节（64 位系统），已通过测试验证。

## 边界问题与解决方案

| 问题 | 解决方案 |
|------|----------|
| 部分游戏不响应虚拟键码 | 使用扫描码（KEYEVENTF_SCANCODE），多数游戏通过 DirectInput/RawInput 读取扫描码 |
| 扩展键发送失败 | 维护 _ExtendedVKs 集合，匹配时附加 KEYEVENTF_EXTENDEDKEY 标志 |
| SendInput 被反作弊拦截 | 用户层 API 限制，需升级到 Interception 驱动方案 |
| 需要管理员权限 | SendInput 向提权窗口发送时需要提权，文档中提醒用户 |

## 废弃方案

| 方案 | 废弃原因 |
|------|----------|
| 使用虚拟键码（wVk）发送 | 多数游戏读 DirectInput 不响应虚拟键码，扫描码兼容性更好 |
| 直接使用 Interception 驱动 | 需额外安装驱动增加使用门槛，SendInput + 扫描码已覆盖多数场景，Interception 作为后续升级方案 |
